---
title: "Row Level Security"
description: "Supabase Row Level Security policies protecting the Kairo database."
---

Row Level Security (RLS) is a PostgreSQL feature that restricts which rows each database user can read or modify. Supabase evaluates RLS policies automatically on every query — neither the Node.js gateway nor the Python microservice needs to add manual `WHERE` filters to enforce data isolation.

All 14 Kairo tables have RLS enabled. Policies use two Supabase helper functions:

- `auth.uid()` — returns the UUID of the currently authenticated user, cast to `INT` to match `users.id`.
- `auth.jwt()->>'role'` — reads the `role` claim from the user's JWT, returning `'coder'` or `'tl'`.

<Note>
Policies are additive for `SELECT`. If a user matches any SELECT policy on a table, they can read the matching rows. A user with no matching policy sees zero rows rather than an error.
</Note>

---

## Enabling RLS

```sql
ALTER TABLE users                    ENABLE ROW LEVEL SECURITY;
ALTER TABLE soft_skills_assessment   ENABLE ROW LEVEL SECURITY;
ALTER TABLE modules                  ENABLE ROW LEVEL SECURITY;
ALTER TABLE moodle_progress          ENABLE ROW LEVEL SECURITY;
ALTER TABLE topics                   ENABLE ROW LEVEL SECURITY;
ALTER TABLE coder_struggling_topics  ENABLE ROW LEVEL SECURITY;
ALTER TABLE complementary_plans      ENABLE ROW LEVEL SECURITY;
ALTER TABLE plan_activities          ENABLE ROW LEVEL SECURITY;
ALTER TABLE activity_progress        ENABLE ROW LEVEL SECURITY;
ALTER TABLE evidence_submissions     ENABLE ROW LEVEL SECURITY;
ALTER TABLE tl_feedback              ENABLE ROW LEVEL SECURITY;
ALTER TABLE risk_flags               ENABLE ROW LEVEL SECURITY;
ALTER TABLE ai_reports               ENABLE ROW LEVEL SECURITY;
ALTER TABLE ai_generation_log        ENABLE ROW LEVEL SECURITY;
```

## Policy reference

### `users`

| Policy name | Command | Who it applies to | USING / WITH CHECK expression |
|---|---|---|---|
| `users_own_profile` | SELECT | Coder | `id = auth.uid()::int` |
| `users_tl_sees_all` | SELECT | TL (and own row) | `auth.jwt()->>'role' = 'tl' OR id = auth.uid()::int` |
| `users_edit_own` | UPDATE | Any user | `id = auth.uid()::int` |

```sql
CREATE POLICY "users_own_profile"
  ON users FOR SELECT
  USING (id = auth.uid()::int);

CREATE POLICY "users_tl_sees_all"
  ON users FOR SELECT
  USING (auth.jwt()->>'role' = 'tl' OR id = auth.uid()::int);

CREATE POLICY "users_edit_own"
  ON users FOR UPDATE
  USING (id = auth.uid()::int)
  WITH CHECK (id = auth.uid()::int);
```

**Effect:** A coder can read and update only their own user row. A TL can read all user rows but can only update their own.

### `tl_feedback`

This table has five policies because both coders and TLs need different access — and coders need a separate UPDATE policy to mark feedback as read without being able to edit the message text.

| Policy name | Command | Who | Expression |
|---|---|---|---|
| `feedback_own_feedback` | SELECT | Coder | `coder_id = auth.uid()::int` |
| `feedback_tl_sends` | SELECT | TL | `tl_id = auth.uid()::int OR auth.jwt()->>'role' = 'tl'` |
| `feedback_tl_create` | INSERT | TL | `auth.jwt()->>'role' = 'tl' AND tl_id = auth.uid()::int` |
| `feedback_tl_edit` | UPDATE | TL | `tl_id = auth.uid()::int AND auth.jwt()->>'role' = 'tl'` |
| `feedback_coder_mark_read` | UPDATE | Coder | `coder_id = auth.uid()::int AND auth.jwt()->>'role' = 'coder'` |

```sql
CREATE POLICY "feedback_own_feedback"
  ON tl_feedback FOR SELECT
  USING (coder_id = auth.uid()::int);

CREATE POLICY "feedback_tl_sends"
  ON tl_feedback FOR SELECT
  USING (tl_id = auth.uid()::int OR auth.jwt()->>'role' = 'tl');

CREATE POLICY "feedback_tl_create"
  ON tl_feedback FOR INSERT
  WITH CHECK (auth.jwt()->>'role' = 'tl' AND tl_id = auth.uid()::int);

CREATE POLICY "feedback_tl_edit"
  ON tl_feedback FOR UPDATE
  USING (tl_id = auth.uid()::int AND auth.jwt()->>'role' = 'tl');

CREATE POLICY "feedback_coder_mark_read"
  ON tl_feedback FOR UPDATE
  USING (coder_id = auth.uid()::int AND auth.jwt()->>'role' = 'coder');
```

**Effect:** Coders can read feedback addressed to them and set `is_read = true`. TLs can read all feedback, send new messages, and edit their own messages.

## Role-based access summary

| Table | Coder can SELECT | Coder can INSERT | Coder can UPDATE | TL can SELECT | TL can INSERT | TL can UPDATE |
|---|---|---|---|---|---|---|
| `users` | Own row | — | Own row | All rows | — | Own row |
| `soft_skills_assessment` | Own row | — | Own row | All rows | — | — |
| `modules` | All rows | — | — | All rows | Yes | Yes |
| `moodle_progress` | Own rows | — | Own rows | All rows | — | — |
| `topics` | All rows | — | — | All rows | Yes | — |
| `coder_struggling_topics` | Own rows | Own rows | — | All rows | — | — |
| `complementary_plans` | Own rows | — | — | All rows | Yes | Yes |
| `plan_activities` | Own plans' activities | — | — | All rows | Yes | — |
| `activity_progress` | Own rows | Own rows | Own rows | All rows | — | — |
| `evidence_submissions` | Own rows | Own rows | — | All rows | — | — |
| `tl_feedback` | Received rows | — | `is_read` only | All rows | Yes | Own sent rows |
| `risk_flags` | Own rows | — | — | All rows | — | — |
| `ai_reports` | — | — | — | All rows | — | — |
| `ai_generation_log` | — | — | — | All rows | — | — |

<Note>
The Python AI microservice uses the Supabase **service-role key**, which bypasses all RLS policies. This is how it writes to `complementary_plans`, `plan_activities`, `risk_flags`, `ai_reports`, and `ai_generation_log` without needing INSERT policies for those tables.
</Note>
